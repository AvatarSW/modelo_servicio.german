2. Refactorización del Código (Criterios a, e)

Tras analizar el código fuente base de la aplicación de la pizzería (Pizza.java, Pedido.java, GestorPedidos.java y Cliente.java), se detectaron deficiencias estructurales críticas asociadas con el "código sucio" (Bad Smells). Se aplicaron técnicas automatizadas de refactorización mediante el IDE para sanear la arquitectura interna sin alterar el comportamiento funcional del sistema.

Clase Bad Smell Detectado Técnica de Refactorización Aplicada Justificación Limpia (Clean Code)
Pizza / Cliente Violación de Encapsulamiento (Atributos públicos expuestos). Encapsular Campos (Encapsulate Field). Se modificó el acceso a privado (private) y se generaron métodos accesores (getters) para proteger el estado interno del objeto.
Pedido / GestorPedidos Uso de Raw Types (Colecciones ArrayList sin tipar) y castings explícitos peligrosos. Parametrizar Colecciones (Generics). Se parametrizaron las estructuras de datos (List<Pizza>). Evita excepciones del tipo ClassCastException en tiempo de ejecución.
Pedido (calcularTotal) Código Complejo / Torre de condicionales anidados (Anti-patrón de Flecha) y Números Mágicos. Simplificación de Condicionales y Extracción de Constantes. Se eliminó la secuencia inútil de comparaciones. Se implementaron constantes semánticas para el umbral y porcentaje de descuento, optimizando la legibilidad.

4. Pruebas Unitarias con JUnit (Criterio b)

Desde la rama Testing se estructuró la suite de verificación automatizada utilizando el framework JUnit. El foco se centró en asegurar que las rutinas lógicas críticas (como la acumulación de precios y la correcta deducción de descuentos) operen sin fallas bajo cualquier entrada de datos.

  • Caso Feliz: Validación de pedidos estándar donde la suma es exacta y no se alcanzan límites promocionales.
  • Caso Límite: Verificación estricta del comportamiento cuando el importe roza o iguala el umbral de los 50€ para asegurar que el descuento matemático se dispara en el momento preciso.

5. Integración Continua (Criterio h)

Tras verificar localmente la estabilidad de las modificaciones, se realizó la fusión (merge) hacia la línea principal main. Con el fin de garantizar que ningún cambio futuro rompa las funcionalidades existentes, se desplegó un flujo automatizado de integración continua mediante GitHub Actions.

Cada vez que el repositorio remoto recibe código, un servidor virtual ejecuta un entorno aislado, compila el código Java e invoca la suite de pruebas unitarias de manera autónoma.

6. Análisis Estático del Código (Criterios c, d)

Para auditar formalmente la calidad técnica final conseguida tras el proceso de refactorización, se sometió la solución al escrutinio de cuatro motores de análisis estático líderes en el sector industrial:

  • Checkstyle: Auditó el formato del código frente a las convenciones de desarrollo de Google. Confirmó la correcta disposición de llaves, indentaciones uniformes y espaciados adecuados.
  • PMD: Analizó la eficiencia del diseño, verificando que no existiese código muerto, variables sin utilizar u optimizaciones prematuras pendientes en las estructuras de bucle.
  • SpotBugs: Buscó patrones de fallos lógicos ocultos o riesgos potenciales en tiempo de ejecución (como accesos indebidos o desbordamientos estructurales). El reporte arrojó cero alertas críticas.
  • SonarQube: Evaluó las métricas globales del proyecto, garantizando un índice de deuda técnica nulo y certificando que el software cumple holgadamente con los umbrales de paso establecidos (Quality Gate: PASSED).
